# Notes — "#16 Evaluating Prompts Across Models"
**Speaker/Guest:** AI That Works · **Venue:** AI That Works · **Type:** podcast · **URL:** https://www.youtube.com/watch?v=OawyQOrlubM

## Summary (3-6 sentences — what it argues, why it matters for agent evals)
Vaibhav (Vibhav/"Vibbob," BAML) and Dex (HumanLayer, 12-Factor Agents) tackle a recurring practical question: when a new model drops, how do you decide whether to swap it in? Their answer is that for most working systems you should *not* swap — the known model is a "known enemy," and the cost of evaluating change is high because almost nobody has automated evals, so you end up doing "vibe evals" which are inherently slow. The bulk of the episode is a live, screen-shared build of a bespoke cross-model "vibe eval" harness in Python/BAML/Streamlit: dump model outputs to disk as JSON, run all models × all test cases async, then render a side-by-side UI to read the actual emails. The core thesis is that "accuracy" is a poorly-defined stand-in for quality that is entirely specific to your business use case, so eval tooling must be bespoke — and the real, hard, slow work is not the AI but building the infra/UI that lets a human (or team) quickly judge what "good" looks like. It matters for agent evals because it concretely shows the 90%-is-data-plumbing reality, why LLM-as-judge is often premature, and how to turn ad-hoc model comparison into reusable internal tooling.

## Key points (6-14 substantive bullets)
- **Default to not swapping.** When a system works, don't update it; the new model is the "unknown enemy" and "the effort of actually seeing if the model will work or not work is not worth it especially if a system isn't broken." Most people now "let someone else vibe check" first.
- **Step-function exceptions.** Real-world swaps that overrode this: GPT-4 → GPT-4o (Dex's SQL-audit-log summarizer pipeline — chose it purely for 2–4x speed/UX despite being "slightly dumber," running many parallel "summarize the summaries" prompts); GPT-4o-mini's order-of-magnitude latency drop; DeepSeek's open-source weights opening new use cases even at lower quality.
- **Personal/private benchmarks are a real practice.** People keep secret prompt sets as a personal benchmark; Simon Willison's "vibe benchmark" is generating an SVG of a pelican on a bicycle from scratch across Nova, Llama 3.3, DeepSeek — explicitly a vibe, not a technical, benchmark.
- **Two-phase intuition before production eval (Vaibhav's method):** (1) deliberately "turn off" your prior about what models can/can't do, push it with a *shorter, more ambiguous* prompt, run 20–30, and vibe-check whether it does harder problems / better domains; (2) only then eval on the actual production task.
- **Eval the model *in a harness*, not just tokens-in/tokens-out.** Don't only test "text in, prompt, what comes out"; zoom out to whether the model is "smart enough in the right harness" — e.g., can Claude Code one-shot a LinkedIn post via multi-turn tool calling (Dex's "social" prompt that downloads whiteboard images and writes posts). Evaluate architectures, not just models.
- **Switching prompts ≈ cost of a new feature, but slower to verify.** Writing a feature is now fast (agents write code); evaluating an AI pipeline is "always going to be really slow because most people don't have automated evals," so you fall back to slow vibe evals — that's the real bottleneck.
- **BDD/Gherkin doesn't transfer to AI.** Ben Stein's PM-in-AI point: you can't pre-specify "given/when/then" because capabilities are emergent — you throw the new model into the product, discover what it can now do, *then* bake that into the spec and write evals.
- **"Accuracy" is a poorly-defined stand-in for quality** and means different things per context: hallucination rate on factual data vs. "makes working code" vs. tone. The 20%-failure tolerance ("can you imagine a REST API failing on 20% of requests?") depends on whether a human is in the loop to fix the bad 20% (vibe coding) vs. a fully automated, harmful-if-wrong path (reproducing regulation text).
- **Define accuracy from the business use case, backwards.** Email example: "hello first name" doesn't matter while a human pastes output into Loops, but the moment it's fully automated, generating `hello first name` as a substitutable variable becomes a hard correctness constraint — "my accuracy constraints and definition have fundamentally changed." Missing bullets is fine; bad tone is failure.
- **The harness build (concrete recipe):** start with one test case, drop in 4o-mini / Sonnet 4 vs Opus 4 to confirm gut instinct; store test data on disk as JSON; load tests by name; nested loop over models × tests; run **async / maximally parallel**; deliberately **not pytest** because eval is exploratory and "much much closer to Jupyter notebooks" than a unit-test framework; use BAML's dynamic client registry to swap model names at runtime.
- **Failure-tolerant runner pattern.** Wrap each unit in `try/except` that's "not allowed to fail" and does its own save-to-disk, because at scale (100–500 runs) model outages happen — design for partial failure, retries.
- **JSON is useless for judging; build the UI.** "JSON files are garbage" — the practical, under-appreciated truth is the infra/UI work is "surprisingly large," mirroring the ML-era rule that "90% of the job is getting the data in the right shape." Built a Streamlit app with side-by-side A/B model dropdowns; the format must match the artifact (render emails as markdown, draw boxes on images for coordinates) — every eval UI is "bespoke and unique."
- **Cheap, high-signal runtime/eval guards** added live: a **regex M-dash / LLM-tell highlighter** (m-dashes, "comprehensive," "undoubtedly" — flag AI-sounding copy; people auto-filter m-dash emails); placeholder/`[bracket]` detection; link validity (fetch every link, expect 200) — "we don't need it to be perfect... just good enough."
- **LLM-as-judge is premature here.** "I don't even know what's going to judge him" — without a clear definition of a great email, a judge isn't helpful; if you did it, you'd then have to eval the judge prompt separately, "this huge meta problem."
- **The moat is taste + the data of what's good.** Amplify Partners (Sarah/Zara) takeaway from 100 AI founders: the eval harness is easy to build like any test system; the value is *generating the data* of what's good vs. not. Echoes the "Brian" episode where the guest shared code and prompts but would not show the evals.
- **Bespoke vs. generic.** Bespoke is fine and easy and gives full flexibility (shadcn-style scaffold-then-customize); but if your use case is generic, off-the-shelf tools (e.g., bounding-box labelers) win — and "if you're building evals for a super generic use case, you're probably already cooked because that shit's been around for five years."
- **Speed of the whole exercise:** from nothing → full test suite → multi-model viewing → working UI in under an hour (with one accidental full code deletion via Cursor along the way).

## What it adds (non-obvious, talk-specific value vs canonical written sources)
This is a *live build* of a cross-model comparison harness, so it surfaces the unglamorous decisions canonical eval write-ups skip: why **async over pytest** (eval is exploratory, notebook-shaped, partial-failure-tolerant — wrap units in non-failing try/except), why **JSON dumps are useless** and the side-by-side rendered UI *is* the eval, and how much of the work is plumbing (model-name escape slashes, JSON-mode serialization, directory structure). It gives a crisp, reusable framing of "accuracy" as a context-dependent stand-in for quality whose definition flips the instant a human leaves the loop (the `hello first name` → Loops automation example). It contributes the concrete **multi-dimensional eval matrix** insight — you're indexing on test case × model × prompt-version × source-episode simultaneously, and there's "no perfect UI" vendor can sell you. And it offers practically useful cheap guards (regex LLM-tell/m-dash highlighter, placeholder detection, link-200 checks) framed as "good enough," plus the honest stance that LLM-as-judge is premature when you can't even define the target. The closing meta-point — the real moat is taste and the hard-coded record of "what good looks like," not the harness — reframes evals as taste capture rather than scoring infrastructure. Also a candid war-story: Cursor silently deleted all the code mid-demo, motivating the HumanLayer "watch the code, don't write it" / spec-review workflow teaser.
