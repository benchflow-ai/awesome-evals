# Notes — "AI Agent Observability & Evaluation (Bonus Unit 2) — Hugging Face Agents Course"

**Author:** Hugging Face (Agents Course team, co-developed with Langfuse) · **URL:** https://huggingface.co/learn/agents-course/en/bonus-unit2/introduction · **Type:** course · **Found:** true

## Summary (3-6 sentences)
This bonus unit of the free Hugging Face Agents Course is a hands-on, notebook-driven curriculum for instrumenting, monitoring, and evaluating AI agents in production. It teaches you to wire OpenTelemetry-based tracing into the `smolagents` framework (via the OpenInference `SmolagentsInstrumentor`) and ship traces to Langfuse, then track token cost, latency, and error traces. It then walks through two complementary evaluation modes: online evaluation (real-user thumbs up/down feedback and near-real-time LLM-as-a-judge scoring of toxicity/correctness) and offline evaluation against a curated benchmark dataset (GSM8K math problems uploaded as a Langfuse dataset to compare model/tool configurations side-by-side). The framing is explicitly practitioner-oriented — "for everyone who wants to bring their agents in front of users" — emphasizing the cost/accuracy/latency tradeoffs and the feedback loop between production signals and offline regression testing. It is conceptually light but operationally concrete: a runnable end-to-end recipe rather than a theory piece.

## Key points (5-12 substantive bullets)
- **Instrumentation stack is fully specified and runnable.** Install `langfuse`, `smolagents[telemetry]`, `openinference-instrumentation-smolagents`, `datasets`, `smolagents[gradio]`, `gradio`; instrument with one line — `SmolagentsInstrumentor().instrument()` — which auto-captures nested spans for every LLM call and tool invocation. No manual span plumbing for the basic path.
- **Langfuse is the backend via OTLP.** Auth uses `LANGFUSE_PUBLIC_KEY` / `LANGFUSE_SECRET_KEY` / `LANGFUSE_HOST` (EU or US regional endpoints) plus `HF_TOKEN` for HF inference; connection is verified with `get_client()` and `auth_check()`. This is a concrete OpenTelemetry → vendor backend pattern that generalizes beyond Langfuse.
- **Traces vs spans defined operationally.** A trace = a complete agent task start-to-finish (one user query); spans = individual steps inside it (an LLM call, a retrieval). This is the mental model the whole unit builds on.
- **Concrete metric taxonomy:** latency (measure both whole-task and per-step to find bottlenecks), cost (LLMs billed per token; assess whether extra calls earn their marginal gain), request errors (build fallbacks/retries across providers), explicit + implicit user feedback (ratings/comments vs rephrasing/retries), and accuracy (requires defining success criteria first).
- **Online eval — user feedback path:** a Gradio chat UI captures thumbs up/down, mapped to scores (1 = like, 0 = dislike) and logged via `langfuse.create_score()`. This is the cheapest production signal and the unit shows the full wiring.
- **Online eval — LLM-as-judge path:** a separate evaluator LLM scores agent outputs in near-real-time for toxicity or correctness, with results logged back as Langfuse scores. Judge templates are prompt-based and run alongside live traffic.
- **Trace enrichment for slicing:** spans get `user_id`, `session_id`, `tags`, and `metadata` via `span.update_trace()`, enabling per-user/per-session debugging — a practical detail often skipped in intro material.
- **Offline eval with GSM8K is the standout exercise.** Load GSM8K (grade-school math with ground-truth solutions) from HF, create a Langfuse dataset with `create_dataset()`, add input/expected-output pairs via `create_dataset_item()`, then link each agent run to its item with the `item.run()` context manager — giving repeatable accuracy metrics because ground truth exists.
- **Config comparison is the point of offline eval:** the notebook runs multiple model/tool configurations (e.g. bare `CodeAgent(tools=[])` vs `CodeAgent` + `DuckDuckGoSearchTool`) against the same dataset to compare side-by-side. A `run_smolagent()` helper wraps execution in `start_as_current_generation()` spans for measurement.
- **Explicit online↔offline loop:** the unit frames the two modes as complementary — offline gives repeatable ground-truth accuracy; online catches unexpected scenarios and model drift the lab misses — and argues successful evaluation blends both in a continuous iteration loop.

## What it adds / why it's good (non-BS practitioner value)
Most agent-eval writing stays at the conceptual level (here's why you need evals, here's what LLM-as-judge is). This unit's value is that it's a *copy-pasteable end-to-end pipeline*: real package names, real env vars, a one-line instrumentor, a Gradio feedback loop, a judge wired to scores, and an actual benchmark (GSM8K) loaded into a dataset entity with a `.run()` harness for A/B-ing configs. For someone standing up agent observability for the first time, it collapses the gap between "I should trace my agent" and "my traces, costs, and accuracy are in a dashboard I can slice by user." It also models the *correct relationship* between observability and eval — traces are the substrate, online judges/feedback are cheap-but-noisy production signals, and offline ground-truth benchmarks are the repeatable regression gate — and insists on combining them rather than picking one. The OpenTelemetry framing means the skills port to any OTLP backend, not just Langfuse. The main caveat: it is a vendor-flavored tutorial (Langfuse + smolagents specific) and light on judge-reliability pitfalls, statistical rigor, or judge-vs-human-correlation — it teaches the plumbing, not the science of trustworthy judging.
