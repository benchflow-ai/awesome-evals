# Notes — "Evaluate agent trajectories"

**Author:** Comet / Opik Documentation · **URL:** https://www.comet.com/docs/opik/evaluation/evaluate_agent_trajectory · **Type:** tool · **Found:** true

## Summary
This is a hands-on Opik (Comet's open-source LLM observability platform) how-to for evaluating an agent's *trajectory* — the sequence of steps, tool calls, and intermediate decisions it takes to reach an answer — rather than only its final output. The core argument is that trajectory evaluation catches tool-selection errors and inefficient reasoning paths before they reach production. The mechanism is a tracing-first workflow: instrument the agent with Opik tracing, build a dataset of inputs with expected tool calls, write a custom metric that reads the captured trajectory via an optional `task_span` parameter, and run `evaluate()` to score each case. The page ships runnable code for two stacks (LangChain via `OpikTracer`, OpenAI via `@track`/`track_openai`) and two reference metrics — "Strict Tool Adherence" (order matters) and "Tool Adherence" (order-agnostic) — that recursively walk the span tree to extract the tools actually used. A companion page documents a `TrajectoryAccuracy` LLM-judge metric scoring ReAct thought/action/observation sequences from 0.0–1.0. The note's value is its concrete bridge from trace instrumentation to programmatic, dataset-driven trajectory scoring.

## Key points
- **Premise: outputs aren't enough.** "Evaluating agents requires more than checking the final output. You need to assess the **trajectory** — the steps your agent takes to reach an answer, including tool selection, reasoning chains, and intermediate decisions." Trajectory eval specifically targets tool-selection errors and inefficient reasoning paths pre-production.
- **Tracing is the prerequisite, not optional.** You can only evaluate a trajectory you captured. The workflow requires the agent to be instrumented with Opik tracing first (`OpikTracer` callback for LangChain; `track_openai()` + `@track` decorators for raw OpenAI). Tool functions are tagged with `@track(type="tool")` so they show up as tool spans.
- **`task_span` is the load-bearing primitive.** Custom metrics get an optional `task_span` parameter (a `SpanModel`) that exposes the full nested span hierarchy of the run. This is how a metric reaches into the actual execution trajectory rather than just the input/output pair — the doc calls it "the key to this metric."
- **Tools are extracted by recursively walking the span tree.** Both reference metrics implement `find_tools()` that recurses through `task_span.spans`, collecting `span.name` wherever `span.type == "tool"`. This means tool detection is structural (from traced spans), not parsed from text.
- **Two flavors of tool-adherence metric.** "Strict Tool Adherence" compares `tool_used == expected_tool` (list equality, so order and multiplicity matter). "Tool Adherence" compares `set(tool_used) == set(expected_tool)` (order-agnostic). Same skeleton, one-line difference — a clean illustration of how eval strictness is a design choice.
- **Metrics are plain `BaseMetric` subclasses returning `ScoreResult`.** Each returns `value=1.0/0.0` plus a human-readable `reason` (e.g. `f"Used {tool_used}, expected {expected_tool}"`), making failures self-explanatory in the UI.
- **Dataset format is deliberately loose.** Items are arbitrary dicts (e.g. `{"input": "What is the weather in SF?", "expected_tool": ["get_weather"]}`); a negative example uses `"expected_tool": []` (a pure-math question that should call no tool — tests over-tooling). "The format of dataset items is very flexible, you can include any fields you want in each item."
- **The run loop is standard Opik `evaluate()`.** You define an `evaluation_task(dataset_item)` that invokes the agent (with the tracer in callbacks) and returns `{"output": ...}`, then pass `dataset`, `task`, `scoring_metrics=[StrictToolAdherenceMetric()]`, and `project_name` to `opik.evaluation.evaluate`.
- **Project-scoping is mandatory in Opik 2.0.** A prominent note warns that datasets and experiments are project-scoped — you must pass `project_name` when creating datasets and running experiments or they won't associate correctly.
- **Companion LLM-judge metric: `TrajectoryAccuracy`.** From the metrics page, `TrajectoryAccuracy` checks how closely a ReAct-style agent followed "a sensible sequence of thoughts, actions, and observations to achieve the stated goal," returning a score "between 0.0 and 1.0" with reasoning. It takes `goal` (str), `trajectory` (list of `{thought, action, observation}` dicts), and `final_result` (str); defaults to judge model `gpt-5-nano`. Positioned for "auditing complex workflow agents and reinforcement-learning traces."
- **Results analysis is trace-linked.** The experiment dashboard scores each case; clicking a row exposes the full agent execution trajectory via a `Trace` button — i.e. score and raw trajectory are joined in one view for debugging.
- **CI/CD and framework hooks exist.** The page points to framework-specific setups (CrewAI, LangGraph, OpenAI Agents) and to PyTest integration, so trajectory evals can run as gates in a pipeline.

## What it adds / why it's good
Most "evaluate your agent" content stops at LLM-as-judge over the final answer. This page is unusual in that it closes the loop between *observability* and *evaluation*: the metric reads the real, traced execution graph (`task_span` → nested spans) instead of re-parsing transcript text, so tool-use checks are grounded in what actually executed. The two near-identical metrics (list-equality vs set-equality) are a quietly excellent teaching device — they make the "how strict should my trajectory check be?" decision concrete in one line of code. The negative test case (`expected_tool: []` for a math question) shows the under-discussed failure mode of *over-tooling*, not just wrong-tool. It's also honest about being instrumentation-first: you cannot evaluate a trajectory you didn't capture. The pairing with the `TrajectoryAccuracy` LLM-judge metric gives two complementary levers — cheap deterministic structural checks (tool adherence) plus a graded judge for reasoning quality — which is the right mental model. Caveat for a knowledge base: it's vendor docs, so it's framed entirely around Opik primitives, and the reference metrics import from `opik.message_processing.emulation.models`, an internal-looking path that may drift across versions.
